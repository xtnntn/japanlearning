# Kotoba Atelier：产品决策记录

> 状态：MVP 设计已确认  
> 第一服务对象：开发者本人  
> 第一终端：Mac 原生桌面应用  
> 本文是后续设计与开发的单一决策依据；未确认的功能不应抢占 MVP 范围。

## 1. 产品定位与目标

这是一个 AI 驱动的个性化日语阅读训练器，而不是翻译网页、资讯流或通用背词 App。

核心目标：逐步不依赖中文翻译，独立读完自己感兴趣的真实日文文章。

首批阅读题材：ACGN、虚拟主播及类似情报资讯/评论内容。

## 2. 每日核心闭环

```text
定时通知当天唯一文章标题
        ↓
在干净阅读器中读完一篇真实日文文章
        ↓
任意划词，先看日语提示，按需展开中文
        ↓
到达文末后自动完成理解题
        ↓
显示题目对应原文证据与题材反馈
        ↓
更新兴趣与能力画像，准备次日文章
```

### 每日文章

- 每天只提供一篇主阅读文章，约 10 分钟。
- 文章是强制训练任务，不因临时兴趣不符而更换。
- 任务最低完成条件：阅读到文章结尾。
- 当天未完成则次日直接过期并记录未完成；不积压补读债务，也不阻断次日任务。
- 每日通知只展示当天文章标题。

### 选文规则

- 第一版唯一内容来源：KAI-YOU。
- 仅处理公开、编辑部原创、正文足够长的文章；过滤会员限定、过短快讯和用户投稿。
- 适配度高于新鲜度：若没有合适新文，使用用户从未读过但高度匹配的历史文章。
- 每周两篇为探索题材文章；其余文章优先来自已验证兴趣。
- 文章结尾提供一道题材反馈单选题：
  - 想多读这个题材；
  - 题材还行；
  - 题材不感兴趣；
  - 不是题材问题，是这篇太难或写得不好。

## 3. 划词解释

- 用户可任意选择词、短语或句子；AI 不替用户预先划重点。
- 第一屏给出简短、受控难度的日语释义或上下文提示。
- 日语释义优先使用系统已确认掌握的词；解释中的汉字使用振假名，例如 `曖昧（あいまい）`。
- 中文翻译仅在用户主动点击后展开。
- 默认解释必须短；更深入讲解只能由用户主动展开。
- 划词、中文展开、阅读完成、题目表现均为兴趣和能力画像的信号。

## 4. 原文与内容边界

- 阅读器在产品内展示干净排版的全文，并保留原始图片、社媒帖文、视频等嵌入和来源回链。
- 系统不重新托管第三方媒体。
- 原文绝不改写、插词或混入 AI 生成句子。
- 第一版只服务本人：文章全文只用于私有、短期预处理和临时阅读视图；不建立公共全文库，不向其他用户再发布。
- 系统可在通知前短期预处理候选全文，用于选文、难度评估和预生成题目；到期删除全文缓存。
- 若未来面向其他用户，必须改用授权、开放许可或合作内容。

## 5. 理解题与能力画像

### 自动理解题

- 读到文末后自动出现少量理解题。
- 题目不作为“完成文章”的通关门槛，而是能力画像证据。
- 每个正确答案必须定位到原文证据句。
- 答题后显示证据句，并简短解释错误选项为什么不成立。

### 初始与周期性评估

- 首次使用进行约 15–20 分钟的阅读与词汇定位测试。
- 该测试建立初始画像，不用于贴永久 JLPT 标签。
- 每周进行一次约 10 分钟的独立阅读评估：使用用户未读过的新文章，评估期间关闭中文翻译和划词解释，结束后通过选择题判断理解。
- 日常辅助阅读数据与无翻译评估数据必须分开记录。

### 首页主指标

首页唯一核心进步指标：**新语境中已学表达的独立理解率**。

只有同时满足以下条件才计为独立理解：

1. 曾经划选过的表达在新文章中再次出现；
2. 用户未展开中文解释；
3. 用户能答对相关理解题。

“可接受文章难度”与划词频率是次级趋势，不替代主指标。

## 6. 两周 MVP 验收

两周试验有效，必须同时满足：

1. 按文章长度与难度归一化后的划词频率下降；
2. 每周无翻译新文章的理解题表现不退步；
3. 新语境中已学表达的独立理解率上升。

单独降低划词频率不算进步，避免通过推荐更简单文章制造假象。

## 7. MVP 范围取舍

两周原型必须保留：

- 每日文章通知与文章选择；
- KAI-YOU 单源文章解析与干净阅读器；
- 任意划词的分层解释；
- 自动理解题、原文证据与结果记录；
- 初始定位、每周独立评估与基础能力画像。

明确延后：

- 内置 Anki 卡片草稿；
- 间隔复习队列；
- AI 复现句；
- 多站点内容接入；
- 追番陪看、音频转写和实时语音。

## 8. 技术框架

| 层 | 技术选择 | 责任 |
| --- | --- | --- |
| 桌面壳 | Tauri 2 | Mac 原生窗口、菜单、通知、权限 |
| UI | React + TypeScript | 阅读器、文本选择、解释浮窗、题目、数据面板 |
| 本地核心 | Rust | 内容获取/解析、缓存清理、LLM 调用、系统调度 |
| 本地数据 | SQLite | 学习画像、文章历史、划词、题目与趋势数据 |
| 密钥 | macOS Keychain | 保存 API Key，前端不可访问 |
| AI | OpenAI Responses API | 解释、难度分析、出题、兴趣与能力画像更新 |
| 通知 | Tauri 通知 + `launchd` helper | App 退出时的定时提醒 |

### 架构原则

- 本地优先：第一版不需要账号系统、云数据库、Docker、FastAPI 或 Supabase。
- React 前端不持有 API Key；由 Rust 从 Keychain 读取密钥并调用 API。
- KAI-YOU 解析先按单站点 DOM 结构实现，不在 MVP 阶段引入泛化爬虫框架。
- 所有 AI 结构化结果都应保存为可验证字段，题目必须保存对应原文证据位置。

## 9. 后续待决问题

- 每日通知的具体时间与用户可配置方式；
- KAI-YOU 的文章解析选择器、嵌入内容与会员内容的实际兼容性；
- 初始定位测试与每周评估的题库规格；
- 能力画像的数据结构、难度归一化方法与模型提示词；
- Mac 应用打包、`launchd` 定时通知和 Keychain 集成的实现验证。
